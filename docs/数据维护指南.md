# 📋 末日生存游戏 — 数据维护指南

## 核心原则

- **游戏数据按模块组织在 `js/data/` 文件夹中**，各子模块负责不同类型的数据
- **`js/config.js` 作为统一的重新导出入口**，所有数据通过 `config.js` 对外暴露，其他文件只需从 `config.js` 导入
- 修改 `js/data/` 下的数据文件后刷新浏览器即可生效
- 其他文件（`combat.js`、`maps.js`、`state.js` 等）不需要修改
- 新增物品只需在对应子模块的数组中添加条目
- `config.js` 和 `utils.js` 内部有些常量是自动派生的（如 `NURSE_MEDICINE_POOL`、`V_TRADE_AMMO_TYPES`、`BUILDING_MATERIAL_NAMES`、`BOSS_NAMES`），修改源数据后自动更新

## 二、常见操作指南

### 2.1 新增食物

在 `FOODS` 数组末尾添加条目即可。

**字段说明：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | string | ✅ | 唯一标识，全局唯一 |
| `name` | string | ✅ | 显示名称 |
| `type` | string | ✅ | 固定为 `"food"` |
| `hunger` | number | ✅ | 恢复饱腹值 |
| `hydration` | number | ❌ | 恢复水分值，省略则不恢复水分 |

**示例：添加"红烧肉"**

```js
// 在 FOODS 数组末尾添加
{ id: "红烧肉", name: "红烧肉", type: "food", hunger: 50, hydration: 10 },
```

添加后，作弊系统自动支持 `/get_food_红烧肉_10` 格式获取。地图掉落中 `food` 类型会自动包含新食物。

---

### 2.2 新增近战武器

在 `MELEE_WEAPONS` 数组末尾添加条目。

**字段说明：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | string | ✅ | 唯一标识 |
| `name` | string | ✅ | 显示名称 |
| `type` | string | ✅ | 固定为 `"melee"` |
| `damage` | number | ✅ | 基础伤害值 |
| `durability` | number | ✅ | 最大耐久度（拳头为 `Infinity`） |
| `rarity` | string | ❌ | 稀有度：`common`/`uncommon`/`rare`/`epic`/`legendary`，拳头无此字段 |
| `comboRate` | number | ✅ | 连击概率（0~1） |

**示例：添加"链锯剑"**

```js
{ id: "链锯剑", name: "链锯剑", type: "melee", damage: 110, durability: 90, rarity: "epic", comboRate: 0.16 },
```

**稀有度与连击率对应关系：**

| 稀有度 | 连击率 | 伤害范围参考 |
|--------|--------|-------------|
| common | 0.05 | 30~48 |
| uncommon | 0.08 | 40~70 |
| rare | 0.12 | 66~90 |
| epic | 0.15 | 99~105 |
| legendary | 0.18 | 118+ |

### 2.3 新增远程武器

添加远程武器需要**同时关注弹药关联**，步骤如下：

**步骤1：在 `RANGED_WEAPONS` 添加武器**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | string | ✅ | 唯一标识 |
| `name` | string | ✅ | 显示名称 |
| `type` | string | ✅ | 固定为 `"ranged"` |
| `damage` | number | ✅ | 基础伤害值 |
| `integrity` | number | ✅ | 完整度（通常为100） |
| `ammoType` | string | ✅ | 使用的弹药类型，必须与 `AMMO` 中的弹药 `id` 对应 |
| `critRate` | number | ✅ | 暴击概率（0~1） |
| `rarity` | string | ✅ | 稀有度 |

**步骤2：如果需要新弹药类型，在 `AMMO` 添加弹药**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | string | ✅ | 弹药唯一标识 |
| `name` | string | ✅ | 显示名称 |
| `type` | string | ✅ | 固定为 `"ammo"` |
| `compatibleWith` | string[] | ✅ | 兼容武器ID列表 |

**步骤3：如果使用已有弹药，在对应弹药的 `compatibleWith` 数组中添加新武器ID**

**示例：添加"SCAR-H"（使用已有弹药 7.62×51mm）**

```js
// 1. 在 RANGED_WEAPONS 末尾添加
{ id: "SCAR-H", name: "SCAR-H", type: "ranged", damage: 85, integrity: 100, ammoType: "7.62×51mm", critRate: 0.28, rarity: "rare" },

// 2. 在 AMMO 中找到 "7.62×51mm"，在 compatibleWith 数组中添加 "SCAR-H"
{ id: "7.62×51mm", name: "7.62×51mm", type: "ammo", compatibleWith: ["M700", "FAL", "EBR14", "SR25", "SCAR-H"] },
```

**示例：添加"SCAR-H"（使用全新弹药 7.62×51mm NATO）**

```js
// 1. 在 RANGED_WEAPONS 末尾添加
{ id: "SCAR-H", name: "SCAR-H", type: "ranged", damage: 85, integrity: 100, ammoType: "7.62×51mm NATO", critRate: 0.28, rarity: "rare" },

// 2. 在 AMMO 末尾添加新弹药
{ id: "7.62×51mm NATO", name: "7.62×51mm NATO", type: "ammo", compatibleWith: ["SCAR-H"] },
```

**暴击率参考（按武器类别）：**

| 武器类别 | 暴击率范围 | 弹药类型 |
|----------|-----------|---------|
| 弓/弩 | 0 | 箭矢 |
| 手枪 | 0.15~0.20 | 9×19mm / .357 Magnum |
| 冲锋枪 | 0.18~0.23 | 9×19mm |
| 步枪(5.56) | 0.22~0.25 | 5.56×45mm NATO |
| 步枪(7.62×39) | 0.20~0.28 | 7.62×39mm |
| 战斗步枪(7.62×51) | 0.22~0.32 | 7.62×51mm |
| 狙击枪 | 0.35~0.45 | 7.62×51mm / .300 Win Mag |
| 霰弹枪 | 0.10~0.12 | 12号霰弹 |

### 2.4 新增/修改背包

修改背包需要同时维护 `BACKPACK_TYPES` 和 `LOOT_BACKPACKS` 两个数组。

**字段说明：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | string | ✅ | 唯一标识，与 `name` 相同 |
| `name` | string | ✅ | 显示名称 |
| `type` | string | ✅ | 固定为 `"backpack"`（`BACKPACK_TYPES` 中为具体背包名） |
| `capacity` | number | ✅ | 背包容量（格位） |
| `rarity` | string | ❌ | 稀有度，仅 `LOOT_BACKPACKS` 需要 |

#### `BACKPACK_TYPES` — 所有背包类型定义

作为背包的完整字典，**所有可用背包**都在此注册。新游戏初始背包也从中选取。

```js
export const BACKPACK_TYPES = {
  口袋: { id: "口袋", name: "口袋", type: "口袋", capacity: 15 },
  // ... 已有背包
};
```

**新增示例：**

```js
// 在 BACKPACK_TYPES 中添加新条目
  轻型战术包: { id: "轻型战术包", name: "轻型战术包", type: "轻型战术包", capacity: 38 },
```

#### `LOOT_BACKPACKS` — 可掉落背包列表

控制哪些背包会作为战利品掉落，以及各自的稀有度权重。

**新增示例：**

```js
// 在 LOOT_BACKPACKS 末尾添加
{ id: "轻型战术包", name: "轻型战术包", type: "backpack", capacity: 38, rarity: "rare" },
```

**⚠️ 注意事项：**

1. **两个数组必须同步**：如果只在 `LOOT_BACKPACKS` 添加而不在 `BACKPACK_TYPES` 添加，新背包在被添加到玩家背包时可能无法被正确识别。
2. **`id` 与 `name` 保持一致**：背包的 `id` 和 `name` 必须相同，这是游戏逻辑的约定。
3. **新增背包自动纳入掉落**：在 `LOOT_BACKPACKS` 中添加后，所有 `lootTable` 中包含 `backpack` 的地图都会随机掉落该背包，无需修改其他文件。
4. **容量建议范围**：

| 稀有度 | 容量参考 |
|--------|---------|
| common | 20~24 |
| uncommon | 28~32 |
| rare | 36~42 |
| epic | 48~56 |

### 2.5 修改BOSS掉落

在 `FIXED_LOOT_DROPS` 中修改对应条目。

**当前固定掉落一览：**

| 键名 | 来源 | 当前掉落 |
|------|------|---------|
| `banker_kill` | 击杀银行行长 | 武士刀（近战） |
| `leader_gift` | 阵地首领赠送 | 唐横刀（近战） |
| `doctor_trade` | 陈博士交易 | M700 + 7.62×51mm×30 |
| `outlaw_kill` | 击杀马三 | AK47 + 7.62×39mm×30 |
| `shadow_kill` | 击杀黑影 | GP100 + .357 Magnum×20 |
| `space_crate` | 航天基地箱子 | AWM + .300 Win Mag×30 |
| `tunnel_cache` | 隧道缓存 | 9×19mm×20 + 压缩饼干 |

**示例：把马三掉落从 AK47 改为 HK416**

```js
outlaw_kill: { weaponId: "HK416", type: "ranged", ammoId: "5.56×45mm NATO", ammoCount: 30 },
```

**示例：把银行行长掉落改为远程武器**

```js
banker_kill: { weaponId: "EBR14", type: "ranged", ammoId: "7.62×51mm", ammoCount: 30 },
```

**注意：** `type` 字段决定从哪个数组查找武器，`"melee"` 从 `MELEE_WEAPONS` 查找，`"ranged"` 从 `RANGED_WEAPONS` 查找。如果掉落包含弹药，`ammoId` 必须与 `AMMO` 中的 `id` 对应。

### 2.7 修改地图掉落概率

在 `MAPS` 的 `lootTable` 中调整权重。权重是相对值，不需要加起来等于100，但习惯上总和为100方便理解。

**lootTable 可用的 key：**

| key | 说明 | 对应数据源 |
|-----|------|-----------|
| `food` | 食物 | `FOODS` |
| `drink` | 饮品 | `DRINKS` |
| `fruit` | 水果 | `FRUITS` |
| `cigarette` | 香烟 | `CIGARETTES` |
| `medicine` | 药品 | `MEDICINES`（按稀有度加权） |
| `backpack` | 背包 | `LOOT_BACKPACKS`（按稀有度加权） |
| `melee` | 近战武器 | `MELEE_WEAPONS`（排除拳头，按稀有度加权） |
| `ranged` | 远程武器 | `RANGED_WEAPONS`（按稀有度加权） |
| `ammo` | 弹药 | `AMMO`（低危地图箭矢概率高） |
| `building` | 建材 | `BUILDING_MATERIALS` |
| `empty` | 无掉落 | — |

**示例：增加"废弃警察局"的远程武器掉落率**

```js
// 原始
{ id: "废弃警察局", name: "废弃警察局", danger: "★★★★高危", encounterRate: 0.55,
```

| `name` | string | 任务名称 |
| `desc` | string | 任务描述 |
| `reqAffection` | number | 所需好感度 |
| `require` | object | 提交需求 |
| `reward` | object | 奖励 |
| `story` | string | 任务剧情文本 |

**require 支持的格式：**

```js
// 食物+饮品
require: { food: 1, drinks: 1 }

// 指定药品
require: { medicine: 5, medicineId: "止血带" }

// 指定饮品
require: { drinks: 10, drinkId: "纯牛奶" }

// 多种物品组合
require: { items: [{ type: "medicine", id: "医用急救包", count: 6 }, { type: "medicine", id: "抗感染血清", count: 6 }] }

// 所有药品各N个
require: { allMedicine: 2 }

// 所有罐头各1个
require: { allCanned: 1 }

// 所有水果各1个
require: { allFruits: 1 }

// 所有饮品+所有食物各1个
require: { allDrinks: 1, allFoods: 1 }
```

**reward 支持的格式：**

```js
// 单个物品
reward: { item: "小匕首", desc: "小匕首×1" }

// 物品+弹药
reward: { item: "P226", ammo: { type: "9×19mm", count: 50 }, desc: "P226×1 + 9×19mm×50" }

// 多个物品
reward: { itemStack: [{ id: "手术包", count: 3 }], desc: "手术包×3" }

// 香烟+弹药
reward: { cigarettes: 20, ammo: { type: "9×19mm", count: 50 }, desc: "随机香烟×20 + 9×19mm×50" }
```

**示例：修改V小姐的v1任务奖励**

```js
v1: { name: "初见任务", desc: "上交1份食物+1份饮品", reqAffection: 0, require: { food: 1, drinks: 1 }, reward: { item: "大扳手", desc: "大扳手×1" }, story: "\"饿着肚子没法战斗。给我吃的，我给你工具。\" 她用扳手敲了敲墙壁。" },
```

## 三、`js/data/` 文件夹结构与扩展指南

### 3.1 文件夹组织原则

`js/data/` 文件夹按数据类型划分子目录，每个子目录包含：

- **数据文件**：定义具体的常量和数据
- **`index.js`**：统一导出该目录下的所有常量

`js/data/index.js` 作为顶层入口，从各子目录的 `index.js` 重新导出所有常量。`js/config.js` 再从 `js/data/index.js` 导入并重新导出，同时提供工具函数。

### 3.2 导出链路

```
数据文件 (如 foods.js)
  → 子目录 index.js (如 items/index.js)
    → js/data/index.js
      → js/config.js (重新导出 + 工具函数)
        → 其他模块 (如 game.js, combat.js 等)
```

### 3.3 如何添加新的数据文件

**步骤1：在对应子目录中创建数据文件**

例如，在 `js/data/items/` 中添加新的 `crops-ext.js`：

```js
// js/data/items/crops-ext.js
export const SPECIAL_CROPS = [
  { id: "变异种子", name: "变异种子", matureTurns: 10, reward: { food: "万能针剂" } },
];
```

**步骤2：在子目录的 `index.js` 中添加导出**

```js
// js/data/items/index.js
export { FOODS, FRUITS } from './foods.js';
export { DRINKS } from './drinks.js';
// ... 已有导出 ...
export { SPECIAL_CROPS } from './crops-ext.js';  // 新增
```

**步骤3：在 `js/config.js` 中导入（如需在工具函数中使用）**

如果新常量需要在 `config.js` 的工具函数中使用，需在 import 语句中添加；否则 `export * from './data/index.js'` 会自动重新导出，无需额外操作。

### 3.4 如何添加新的子目录

如果需要创建全新的数据分类（如 `js/data/quests/`）：

**步骤1：创建子目录和文件**

```
js/data/quests/
  ├── index.js
  └── main-quests.js
```

**步骤2：在 `js/data/index.js` 中添加导出**

```js
export * from './quests/index.js';
```

**步骤3：在子目录的 `index.js` 中导出常量**

```js
// js/data/quests/index.js
export { MAIN_QUESTS } from './main-quests.js';
```

### 3.5 注意事项

1. **所有常量最终通过 `config.js` 导出**：其他模块应从 `config.js` 导入，而非直接从 `js/data/` 子目录导入
2. **`utils.js` 中的派生常量**：`CANNED_FOOD_IDS`、`TOOL_WEAPON_IDS`、`V_TRADE_AMMO_TYPES` 等派生常量定义在 `js/data/utils.js` 中，它们依赖其他子模块的数据
3. **新增子目录后**：务必在 `js/data/index.js` 中添加 `export *` 语句，否则常量不会被导出
4. **文件命名**：使用小写字母和连字符（如 `building-materials.js`、`map-actions.js`）

### 掉落权重说明

掉落权重用于 `pickRandomLoot` 函数中的加权随机选取。权重越高，出现概率越大。实际概率计算方式：

```
某稀有度概率 = 该稀有度权重 / 所有稀有度权重总和
```

| 稀有度 | 权重 | 大致概率 |
|--------|------|---------|
| common | 20 | 47.6% |
| uncommon | 12 | 28.6% |
| rare | 6 | 14.3% |
| epic | 3 | 7.1% |
| legendary | 1 | 2.4% |

## 五、数据关联图

以下常量之间存在引用关系，修改时需注意联动：

### 弹药 ↔ 武器（双向关联）

```
AMMO.compatibleWith[]  ←→  RANGED_WEAPONS.ammoType
```

- 每把远程武器的 `ammoType` 必须在 `AMMO` 中有对应条目
- 每种弹药的 `compatibleWith` 必须包含所有使用该弹药的武器ID
- **新增远程武器时**：如果使用已有弹药，需在弹药的 `compatibleWith` 中添加武器ID
- **新增弹药时**：需确保所有使用该弹药的武器都在 `compatibleWith` 中

### 固定掉落 → 物品数组

```
FIXED_LOOT_DROPS.weaponId  →  MELEE_WEAPONS / RANGED_WEAPONS
FIXED_LOOT_DROPS.ammoId    →  AMMO
FIXED_LOOT_DROPS.foodId    →  FOODS
```

- `type` 字段为 `"melee"` 时，`weaponId` 从 `MELEE_WEAPONS` 查找
- `type` 字段为 `"ranged"` 时，`weaponId` 从 `RANGED_WEAPONS` 查找

### 默认物品 → 各物品数组

```
DEFAULT_ITEM_IDS.melee  →  MELEE_WEAPONS
DEFAULT_ITEM_IDS.food   →  FOODS
DEFAULT_ITEM_IDS.drink  →  DRINKS
DEFAULT_ITEM_IDS.seed   →  BUILDING_MATERIALS
DEFAULT_ITEM_IDS.serum  →  MEDICINES
```

### 自动派生关系

```
NURSE_MEDICINE_POOL  ←  MEDICINES.filter(rarity === "common" || rarity === "uncommon")
V_TRADE_AMMO_TYPES   ←  AMMO.filter(id !== "箭矢").map(id)
BUILDING_MATERIAL_NAMES  ←  BUILDING_MATERIALS 遍历生成 {id: name}
BOSS_NAMES           ←  NAMED_NPCS 遍历生成 name 数组
```

修改源数据后，这些派生常量会自动更新，无需手动修改。

## 六、注意事项

### 1. id 必须全局唯一

每个物品的 `id` 在其所属数组内必须唯一。建议跨数组也保持唯一，避免混淆。

### 2. 修改 id 会影响存档兼容性

玩家的存档中保存的是物品 `id`，如果修改了某个物品的 `id`，旧存档中引用该 `id` 的数据将无法正确匹配。**如需改名，建议只修改 `name` 字段，保持 `id` 不变。**

### 3. 新增物品不需要修改其他文件

在对应数组中添加条目即可，掉落系统、作弊系统等会自动识别新物品。

### 4. 删除物品需要检查所有引用

删除物品前，必须检查以下位置是否引用了该物品：
- `FIXED_LOOT_DROPS` 中的 `weaponId`、`ammoId`、`foodId`
- `DEFAULT_ITEM_IDS` 中的各字段
- `AMMO` 中的 `compatibleWith`
- `TRADER_WEAPON_SHOP` 中的 `weaponId`
- `SURVIVOR_NPC` 中任务的 `require` 和 `reward`
- `LILI_REWARD_MEDICINE_IDS` 和 `XIAOHAN_REWARD_FOOD_IDS`
- `CANNED_FOOD_IDS`
- `TOOL_WEAPON_IDS`
- `TRADE_TEMPLATES` 中的 `ammoType`

### 5. rarity 字段必须是5个值之一

有效的稀有度值：`"common"`、`"uncommon"`、`"rare"`、`"epic"`、`"legendary"`。如果使用了不在这5个值中的稀有度，掉落系统中的加权随机将使用默认权重1，可能导致非预期行为。

### 6. 弹药 compatibleWith 必须与武器 ammoType 对应

- 每把远程武器的 `ammoType` 必须在 `AMMO` 中存在
- `AMMO` 中每种弹药的 `compatibleWith` 必须包含所有使用该弹药的武器
- 两边必须保持同步，否则会导致弹药无法装填或武器无法射击

### 7. FIXED_LOOT_DROPS 中的 type 决定查找范围

- `type: "melee"` → 从 `MELEE_WEAPONS` 查找 `weaponId`
- `type: "ranged"` → 从 `RANGED_WEAPONS` 查找 `weaponId`
- 如果 `type` 与实际武器类型不匹配，将找不到对应武器

### 8. 地图ID与丧尸池映射

`getRandomZombie` 函数使用地图 `id` 精确匹配丧尸池。新增地图时务必在 `poolMap` 中添加映射，否则默认使用低危丧尸池。

### 9. 作弊命令格式

作弊系统使用物品类型和ID生成命令，格式为 `/get_{type}_{id}_{count}`。其中 `type` 对应物品的 `type` 字段值。例如：
- `/get_food_压缩饼干_5`
- `/get_melee_武士刀_1`
- `/get_ranged_AK47_1`
- `/get_medicine_抗生素_3`

### 10. 皇家币系统

皇家币（`royal_coin`）是一种独立货币，不占用背包空间，存储在 `state.royalCoins` 中。
- 通过 `addItem({ type: "royal_coin", count: N })` 添加
- 通过 `removeRoyalCoins(N)` 扣除
- 旧存档通过 `normalizeState()` 自动补全 `royalCoins: 0`

### 11. 汽油系统

汽油是一种可堆叠的特殊货物，不占用背包格子，存储在 `state.gasoline` 中。
- 通过 `addGasoline(N)` 添加
- 通过 `removeGasoline(N)` 扣除，返回实际扣除数量
- 通过 `addItem({ type: "gasoline", count: N })` 也可添加（内部调用 `addGasoline`）
- 通过 `removeItem("gasoline", N)` 也可扣除（内部调用 `removeGasoline`）
- 旧存档通过 `normalizeState()` 自动补全 `gasoline: 0`
- 作弊码：`/get_gasoline_汽油_5`（每次获得5个）
- 汽油在UI货物面板中与香烟、皇家币并列显示
- 汽油支持丢弃和NPC送礼功能

### 12. 剧情专属武器

部分武器仅通过特定剧情获得，不可从掉落池随机获取：
- **原野之弓**（沐苗苗任务三奖励）— `pickRandomLoot` 中排除
- **皇家短枪**（皇后杂货铺购买）— `pickRandomLoot` 中排除
- **皇家银剑**（皇后杂货铺购买）— `pickRandomLoot` 中排除
- **无畏之刃**（新势力结局奖励）— `pickRandomLoot` 中排除

### 13. 交互设计规范

新增或修改NPC交互时，需遵循以下规范：

1. **禁止短路返回**：NPC交互入口函数不得使用 `if (!condition) { setStory(); refreshMenu(); return; }` 模式。应始终显示NPC对话页面和至少一个"返回"选项，条件不满足时只显示对话文本+返回选项。

2. **崩溃值100%行为**：崩溃值满时，玩家无法战斗、进食、饮水、睡觉。拒绝进食/饮水时应提示可用医疗物品（止痛片/肾上腺素等）恢复精神。崩溃满时只允许使用降低崩溃值的药品。

3. **交易背包空间预检查**：交易系统在扣除玩家物品前应检查背包是否有足够空间容纳奖励物品，空间不足时拒绝交易。

4. **危险操作警告**：不可逆选择（如马三任务4）和高风险操作（如食用杂肉+10感染）应显示⚠警告。

5. **禁用选项反馈**：点击 disabled 选项时应显示"条件未满足"提示，而非静默无响应。

6. **消耗品使用后返回**：使用食物/饮品/医疗后应返回对应选择界面而非主菜单，方便连续使用。

7. **黑市商人弹药数量选择**：购买弹药时支持选择数量（1/5/10/全部香烟），不再一次性消耗所有香烟。

8. **柳如烟救援提醒**：发现柳如烟后，生存笔记自动添加救援倒计时。`state.liuruyanDiscovered` 字段追踪是否已发现。

9. **新生力量进度追踪**：完成任意新生力量前置条件后，生存笔记自动添加进度条目（X/6）。

10. **火箭希望结局好感度进度**：选择"带上重要的人"时，显示V/小寒/莉莉丝当前好感度进度。
